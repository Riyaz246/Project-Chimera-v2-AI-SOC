## ğŸ“¸ Technical Pipeline Walkthrough

### Phase 1: Detection Engineering (The Trap)
Instead of relying on fragile file hashes, I engineered a behavioral detection rule in XML. This rule monitors Sysmon Event ID 11 (File Creation) for rapid encryption anomalies (e.g., `.locked` extensions) and maps them to **MITRE ATT&CK T1486**.

![Custom Detection Rule](Screenshot%202025-11-26%20123014.png)
*Figure 1: Custom Wazuh XML rule defining the ransomware behavior logic.*

### Phase 2: The Attack & Automated Response (SOAR)
I developed a Python script to simulate a live ransomware attack. As seen below, the moment the script attempts to encrypt files, Wazuh detects the anomaly and triggers an **Active Response** script (`kill_ransomware.cmd`) that terminates the process ID immediately.

![Attack Simulation](Screenshot%202025-11-26%20132220.png)
*Figure 2: Split-screen view. Left: Ransomware simulator attempting encryption. Right: Wazuh capturing the event and triggering the kill switch.*

### Phase 3: AI-Powered Analysis (The Forensics)
Once the threat is contained, a custom Python listener forwards the alert metadata to a local **Llama 3** model (via Ollama). The AI acts as a Level 3 Analyst, generating a hypothesis, forensic recommendations, and an executive summary.

![AI Incident Report](Screenshot%202025-11-26%20134402.png)
*Figure 3: Automated Incident Report generated by Llama 3, mapped to the specific alert context.*
